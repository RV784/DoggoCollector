# DoggoCollector — Project Context for Claude

This file exists so a fresh Claude Code session (new chat, cleared context window) can pick up this project with zero prior conversation history and not get confused. Read this fully before making changes.

## What this app is

An iOS app (SwiftUI, iOS 27 target) where you point your camera at a real dog you meet, "catch" it, and it becomes a collectible card in your personal album ("Your Pack"). First app in a planned "Collector" family (Doggo → Cat → Flower), each with its own subject/hook, sharing a common Core/DesignSystem layer.

**Visual identity: "Sunny Fetch"** — marigold + cream palette, rounded storybook-ish shapes, a mascot named **Scout** (a simple-shapes dog head), playful/bouncy tone. Not calm/boutique/minimal — the design brief explicitly pushes against "generic AI template" looks (warm-cream+serif, near-black+neon-accent, newspaper-with-hairlines).

## Key product decisions already made (don't re-litigate these)

1. **No walk-streak mechanic.** The build spec originally said Doggo's hook was a walk-streak, calling it "already decided." But the actual design process (visible in the design tool's chat log) shows the user explicitly told Claude Design to drop streak for "collector-first... just capture real dogs," with sharing as a must-have. Confirmed with the user directly in this project too. The home screen is "Your Pack" — a collection grid + catch stats (caught/breeds/today) — not a streak dashboard. `Mechanic/PackCollectorMechanic.swift` implements this.
2. **Typography: SF Rounded, not Baloo 2 / Nunito Sans (for now).** The design's actual brand fonts are Baloo 2 (display) + Nunito Sans (body), both Google Fonts. Font embedding was attempted (downloaded both variable-font files, inspected their `fvar` tables with fontTools) but abandoned mid-implementation at the user's request ("if font is getting tricky, skip for now, we'll add fonts in phase 2"). Currently `DesignSystem/Typography.swift` uses `.system(size:weight:design: .rounded)` everywhere via the `DoggoFont`/`DoggoTextStyle` enums. **Phase 2 TODO**: re-embed Baloo 2 + Nunito Sans. Notes for that work: Baloo 2's variable font has clean named-instance PostScript names (`Baloo2-Regular/Medium/SemiBold/Bold/ExtraBold`) so `Font.custom(name:size:)` should work directly once registered via `CTFontManagerRegisterGraphicsFont`. Nunito Sans's variable font does NOT have per-instance PostScript names (multi-axis: wght/wdth/opsz/YTLC) — that one needs the `UIFontDescriptor` + `.traits: [.weight: ...]` approach instead, or find static per-weight files.
3. **No custom breed classifier.** `DogDetector` (Vision `VNRecognizeAnimalsRequest`) only confirms "a dog is somewhere in this photo" — it cannot identify breed. Per user feedback, catches get a **default generated name that is editable**, not silently assigned forever. See `Mechanic/CatchNameGenerator.swift` (name/breed-label/traits picked from curated word lists) and the "Rename" affordance in both `CatchCelebrationView` and `CardDetailView` (tap pencil icon → alert with a TextField → mutates `dog.name` directly, autosaves via SwiftData).
4. **The share "second person collects this dog into their account" viral loop is NOT implemented.** Phase 1 auth is local-only (no backend, no server accounts), so there's nowhere for a second person's tap to land. `Core/Sharing/ShareLinkBuilder.swift` builds a `doggocollector://catch/<uuid>` deep link and a display string purely for the outgoing share (caption, copy-link, Instagram Story sticker, native share sheet) — the receiving end is a Phase 2 item alongside Firebase + Sign in with Apple/Google.
5. **Camera morph must NOT be full-screen, must use the "peer-layer" matchedGeometryEffect pattern, and now extends all the way into the Gotcha card.** Initial build made the camera cover the entire screen (matching a literal reading of the design chat's "morphs into a full-screen rounded viewfinder" line); a first fix made it a contained bottom panel but applied `matchedGeometryEffect`/`clipShape` directly to the whole `CameraView` (including its live `AVCaptureVideoPreviewLayer`) driven by manually-tracked `topRadius`/`bottomRadius` `@State` — the user tested this and said the morph "looks very bad." The user then supplied a working reference SwiftUI file (a ChatGPT-camera-menu clone) demonstrating the correct technique, and the fix was rebuilt to match it exactly, then **extended one step further** at the user's request so the shutter tap continues the same morph straight into the caught dog's card, rather than cutting to a `.fullScreenCover`.

   **The pattern**: matchedGeometryEffect goes on a trivial background shape (`Color.black`/`DoggoColor.marigold`/the card itself + `.clipShape(...)`) — never on complex UIKit-backed content. Each state is its own small "box" view with a **fixed** frame/shape for that state; matchedGeometryEffect interpolates the frame automatically between whichever two fixed geometries are adjacent in a transition, so there's no need to track a continuous corner-radius value by hand. Actual content (icon+text for the pill; the whole live `CameraView` for the panel) sits as a **separate peer layer inside the same ZStack**, tagged only with `.transition(.opacity)` — it crossfades independently of the geometry animation instead of being dragged through it, which is what made the UIKit camera preview look broken originally. Each conditional box is wrapped in its own stable `ZStack { if condition { box } }` anchor to stop SwiftUI sliding it in from a screen edge on appearance. Animation curve: `.spring(response: 0.4, dampingFraction: 0.8, blendDuration: 0)`, taken directly from the reference.

   **The chain is now 3 states, not 2**: `CollectionView` has a private `SurfaceState` enum (`.idle` / `.camera` / `.celebration`), replacing the old `isCameraActive: Bool` + `.fullScreenCover(item: $justCaught)`. `CatchCelebrationView` is no longer presented modally — it's rendered as a third peer anchor inside `CollectionView`'s own `ZStack`, because **matchedGeometryEffect cannot cross a `.fullScreenCover`/`.sheet` presentation boundary** (a modal is a separate view-controller/window context; the shared `Namespace.ID` has no continuity into it). `CatchCelebrationView` now takes `morphNamespace: Namespace.ID` and `onAddToPack: () -> Void` as parameters instead of reading `@Environment(\.dismiss)`, and applies `.matchedGeometryEffect(id: "catchSurface", in: morphNamespace)` directly to its `DoggoCardView` (unlike the camera panel, the card's own SwiftUI content is simple enough — no UIKit bridging — to be the geometry-matched element directly, rather than needing a separate synthetic background layer). `CameraViewModel.attemptCatch`'s success callback (`CameraView.onCaught`) now flows into `CollectionView.handleCaught(_:)`, which sets `caughtDog` and animates `surfaceState` straight to `.celebration` — the camera panel is never explicitly closed first, so the black rounded rect morphs directly into the white card rect in one continuous animated transaction. "Add to pack" calls `onAddToPack` → `returnToIdle()`, which reuses the same shared id to morph the card back down into the pill, symmetrically, for free. See `CollectionView.swift`'s `SurfaceState`, `catchButton`, `cameraPanel`, `handleCaught`, `returnToIdle`, and `CatchCelebrationView.swift`.
6. **iPhone-only, portrait-only.** `TARGETED_DEVICE_FAMILY = 1`, `INFOPLIST_KEY_UISupportedInterfaceOrientations_iPhone = UIInterfaceOrientationPortrait` in the pbxproj build settings (both Debug/Release, main target). iPad support was deliberately removed from the default Xcode template.

## Architecture

Single Xcode target, plain folders (no SPM package — deliberately deferred until Cat/Flower work starts and it's clear which folders are truly species-agnostic). Uses Xcode's `PBXFileSystemSynchronizedRootGroup` — **any file/folder you create under `DoggoCollector/` is automatically picked up by the build, no pbxproj editing needed for source files.** You only need to touch `project.pbxproj` for build settings (Info.plist keys, device family, etc.).

```
DoggoCollector/
  App/
    DoggoCollectorApp.swift   — @main, builds the SwiftData ModelContainer, constructs UsernameAuthProvider, injects both via .environment()
    AppConfig.swift            — static per-app config (subjectType/tenantId/theme) — the seam Cat/Flower will each override
    RootView.swift             — top-level router: Launch → Landing → Onboarding (if no local UserProfile) else straight to CollectionView
  Core/
    Camera/CameraService.swift       — AVCaptureSession wrapper (session lifecycle + photo capture). No live-frame detection (see Detection below for why).
    Detection/
      SubjectDetecting.swift         — protocol: `detectSubject(in image: CGImage) async -> Bool`. THE per-species seam #1.
      DogDetector.swift               — concrete impl, Vision VNRecognizeAnimalsRequest, confidence threshold 0.6, looks for label "Dog"
    Persistence/
      CaughtDog.swift                 — @Model: id, name, breedLabel, traits([String]), imageData (Data?, .externalStorage), caughtAt, locationLabel, latitude/longitude (coarsened), serialNumber, isFavorite
      UserProfile.swift                — @Model: id, username, createdAt
    Audio/WhistlePlayer.swift         — procedurally synthesizes a two-chirp whistle via AVAudioEngine (no bundled audio asset), mirrors the original design's Web Audio implementation
    Location/
      LocationProvider.swift          — CLLocationManager wrapper (authorization + one-shot current location)
      LocationTagger.swift             — reverse-geocodes + rounds coordinates to ~3 decimal places (~100m) so nothing ever stores/shows an exact address
    Sharing/
      CardRenderer.swift               — `ImageRenderer`-based SwiftUI-view-to-UIImage rasterizer, used to produce the shareable card image
      ShareLinkBuilder.swift           — builds the outgoing share caption + deep link (see decision #4 above)
    Auth/
      AuthProviding.swift              — protocol: currentUsername, signUp(username:), updateUsername(_:), signOut(). THE per-phase seam — phase 2 (Sign in with Apple/Google + Firebase) will be a new conformance, not a rewrite.
      UsernameAuthProvider.swift        — phase-1 concrete impl, @Observable, SwiftData-backed, no backend
  Mechanic/
    CollectorMechanic.swift            — protocol: homeTitle, greeting(username:now:), stats(for:) -> [PackStat]. THE per-species seam #2 (owns the home-screen hook).
    PackCollectorMechanic.swift         — Doggo's concrete impl: collection-stats-only (caught/breeds/today), no streak state at all
    CatchNameGenerator.swift            — curated name/breed-label/trait word lists for the no-classifier default-name flow (decision #3)
  Features/
    Onboarding/  LaunchView, LandingView, OnboardingView    — pre-auth flow
    Collection/  CollectionView, EmptyStateView              — "Your Pack" home screen + empty state
    Camera/      CameraView, CameraViewModel, CameraPreviewView, ViewfinderBracket  — the catch flow
    Card/        CatchCelebrationView, CardDetailView, AmbientBackgroundShapes      — "Gotcha!" + card detail
    Share/       ShareView, ActivityView                     — dark-themed share sheet
    Map/         MapView                                      — MapKit pins at coarsened locations
    Profile/     ProfileView                                  — username + stats + entry to Settings and (currently) nothing else
    Settings/    SettingsView                                  — username edit, notifications toggle (local-only, no backend), location permission status, help/contact mailto, delete account
  DesignSystem/
    Colors.swift        — DoggoColor enum, hex-based (see palette below)
    Typography.swift    — DoggoFont / DoggoTextStyle (SF Rounded — see decision #2)
    Metrics.swift        — DoggoSpacing (4/8/12/16/24/32/40) and DoggoRadius (pill=999, card=28, control=16)
    ScoutMascot.swift    — the mascot, built from SwiftUI shapes (Ellipse head, two ear Ellipses, muzzle, eye Circles, nose, Capsule collar), `ScoutExpression` enum (.idle/.happy/.sad) parameterizes mouth/eyes. Also defines `.floatingIdle()` — a reusable ViewModifier for the continuous subtle up/down drift used on Landing's and Empty-state's mascot.
    Components/
      PillButton.swift    — PillButton (primary marigold / secondary white), TextLinkButton (underlined text CTA), ScalePressButtonStyle (press-down scale feedback used everywhere)
      TagChip.swift        — TagChip (breed/trait pills), StatChip (Collection's caught/breeds/today pills)
      DoggoCardView.swift  — the reusable collectible-card view (photo + breed tag + name + traits + serial badge), `isCompact` toggles between grid-thumbnail and full layouts, `placeholderSeed` picks one of 4 pastel placeholder tints so a grid of un-photographed cards doesn't look monotone. Also defines PolkaDotPlaceholder.
```

`SubjectDetecting` and `CollectorMechanic` are the **only two** things a Cat/Flower app would need to replace — everything else (Core/, DesignSystem/, Features/) is meant to be reusable as-is once extracted into a shared local package (deliberately not done yet, per the build spec's guidance to wait until there's a second app to inform the package boundary).

## Screen-by-screen behavior (as currently built)

1. **Launch** (`LaunchView`) — marigold→marigoldDark gradient, Scout idle (scale-in + fade-in on appear), "DoggoCollector" wordmark + "catch every good dog" tagline, auto-advances to Landing after 1.6s via `.task`.
2. **Landing** (`LandingView`) — cream bg, Scout idle with `.floatingIdle()`, "Catch every good dog you meet." headline, subtext, "Get started" primary CTA → Onboarding, "I already have a pack" text link → **also** Onboarding (there's nothing to restore in phase-1 local-only auth; this is a placeholder for the phase-2 sign-in path, see `RootView.swift` comment).
3. **Onboarding** (`OnboardingView`) — **not verified against real design** (see Source Documents note above). Cream bg, floating Scout, "What should we call you?" headline, an "@username"-styled input (marigold "@" prefix + TextField in a white capsule with shadow — echoes the "@scout" badge motif seen on the real Share screen), "Let's go" CTA disabled until non-empty. Calls `authProvider.signUp(username:)` then `onComplete()` (though `onComplete` is currently a no-op — the screen transition actually happens automatically because `RootView` observes `authProvider.currentUsername` going non-nil and switches to `CollectionView`).
4. **Empty state** (`EmptyStateView`) — shown by `CollectionView.content` when `catches.isEmpty`. Scout `.sad` expression, 70% opacity, `.floatingIdle()`. Does **not** have its own CTA button (see decision-adjacent fix below) — the "Catch a doggo" button is a single persistent element owned by `CollectionView`, always pinned to the bottom regardless of empty/populated state, so there's only ever one dominant action and only one `matchedGeometryEffect` anchor for the camera morph.
5. **Camera** (`CameraView` + `CameraViewModel`) — presented as a **contained bottom panel**, not full screen (decision #5). Live `AVCaptureSession` preview (`CameraPreviewView`, a `UIViewRepresentable` wrapping `AVCaptureVideoPreviewLayer`) with a black fallback when unauthorized/no hardware. Top bar: "● LIVE" badge + Whistle button (replays the chirp via `CameraViewModel.replayWhistle()`). Center: `ViewfinderBracket` (custom corner-mark `Shape`) + "Point at a doggo" / "No dog spotted — try again!" text. Bottom bar: back chevron (`onClose`), shutter (68pt white circle, disabled while `viewModel.isCapturing`), inert "…" options button (not wired to anything — flip-camera would be the natural feature here if picked up later). **No live continuous detection** — detection only runs once, on the captured still frame, after the shutter is tapped (see `CameraViewModel.attemptCatch`); this was a deliberate simplification to avoid fragile cross-actor CVPixelBuffer streaming, and it matches what the real static design screenshots showed anyway (no reactive viewfinder tinting in the actual design).
6. **Catch celebration** (`CatchCelebrationView`) — presented as an **inline peer state of `CollectionView`** (see decision #5), not a `.fullScreenCover` — this is what lets the camera panel morph directly into the card. Marigold→marigoldDark gradient, `AmbientBackgroundShapes` (4 sparse shapes — rounded square, 2 dots, a diamond — that drift continuously via a shared `drift` boolean + per-shape animation `delay`, NOT a one-shot confetti burst; this was corrected after initial build used a dense one-shot 18-piece burst that didn't match the reference), Scout `.happy` with a continuous gentle rock animation, "Gotcha!" + tappable "{name} joined your pack ✏️" (opens the rename alert), the `DoggoCardView` **tilted -3°** and carrying `.matchedGeometryEffect(id: "catchSurface", in: morphNamespace)`, "Add to pack" (calls the passed-in `onAddToPack` closure, morphing the card back down into the pill) / "Share instead" (opens `ShareView` as a sheet) pinned to the bottom via a `ZStack(alignment: .bottom)` + `ScrollView` structure (content scrolls, buttons don't move).
7. **Collection / "Your Pack"** (`CollectionView`) — the home screen and the most structurally important file (owns the camera-morph state). `GOOD MORNING/AFTERNOON/EVENING, {username}` eyebrow + "Your Pack" title + **one** circular avatar button (→ Profile; there is deliberately no second Map icon here anymore, see fix history below) in the header. Stat chips (caught/breeds/today) from `PackCollectorMechanic`. Grid (2-col `LazyVGrid`) of `DoggoCardView(isCompact: true)`, or `EmptyStateView` when empty. "Catch a doggo" button always pinned to the bottom (see Empty-state note above). Owns a private `SurfaceState` enum (`.idle` / `.camera` / `.celebration`) as `@State surfaceState`, plus `@Namespace morphNamespace` and `@State caughtDog: CaughtDog?`, and implements the full 3-state morph via the **peer-layer pattern** (see decision #5): `catchButton`, `cameraPanel`, and the inline `CatchCelebrationView` anchor are each their own stable `ZStack { if surfaceState == .x {...} }` block layered in the same outer `ZStack(alignment: .bottom)`. `catchButton`/`cameraPanel` each pair a simple `matchedGeometryEffect`-tagged background shape with a `.transition(.opacity)` content layer, with its own fixed frame/clipShape radius — pill: `RoundedRectangle(cornerRadius: DoggoRadius.pill)`, height 56; panel: `RoundedRectangle(cornerRadius: 32)` (all four corners, not just the top two — it floats with the same `DoggoSpacing.lg` horizontal/bottom padding as the idle pill, rather than sitting flush against the screen edges, so square bottom corners would look wrong), height `UIScreen.main.bounds.height * 0.62`. The Collection content behind blurs (`.blur(radius: surfaceState == .camera ? 6 : 0)`) only while the camera panel is up, and is also given `.allowsHitTesting(surfaceState == .idle)` plus an invisible `Color.clear.contentShape(Rectangle()).onTapGesture(perform: closeCamera)` scrim shown only in `.camera` state — blur alone is only a visual effect and does NOT block touches, so without this, tapping the dimmed margin around the panel used to fall through and activate whatever card/button was underneath instead of dismissing the camera (the celebration state needs no such scrim since it's a full opaque takeover, nothing dim/blurred to tap through). `openCamera()`/`closeCamera()`/`handleCaught(_:)`/`returnToIdle()` just set `surfaceState` (and `caughtDog` for the catch) inside `withAnimation(morphAnimation)` — no manual corner-radius state tracking anywhere.
8. **Card detail** (`CardDetailView`) — cream bg, custom top bar (back chevron + heart favorite toggle — **see Known Issues, these two buttons currently don't respond to taps**), full `DoggoCardView`, caption ("#0XX in your pack · caught at {location}"), tappable "✏️ Rename" link, "Share this doggo" CTA pinned near the bottom via `Spacer()`.
9. **Share** (`ShareView`) — dark theme (`#1C1712` bg), matches the real design screenshot closely: X close button top-right, "Share {name}" + "Pick a card back", a row of 4 small circular colorway swatches (marigold/sky/peach/ink — tapping one re-renders the card preview with that color as a 6pt border), the card preview itself (compact `DoggoCardView` wrapped in a colored-border overlay), a white metadata block below it ("@{username}" dark pill + 📍 location, then an app-icon + "DoggoCollector — Catch Real Dogs" + a fake `doggocollector.app/{username}/{dogname}` link line), then "Share to Story" (writes the rendered PNG to the pasteboard under the Instagram-stories sticker key and calls `UIApplication.shared.open("instagram-stories://share")`, falling back to the native share sheet if that fails), and a row of three plain-text actions: "Save photo" (`UIImageWriteToSavedPhotosAlbum`, needs `NSPhotoLibraryAddUsageDescription` — already added to the pbxproj), "Copy link", "Share" (native `UIActivityViewController` via `ActivityView`).
10. **Map** (`MapView`) — MapKit `Map`/`Annotation` per caught dog at its coarsened lat/long, custom marigold paw-print pin, tap → `CardDetailView` as a sheet. Reached from `ProfileView`... **actually currently NOT linked from anywhere in the nav graph** — the old Collection-header map icon was removed as part of matching the real design (which only shows one avatar icon), and a replacement entry point inside `ProfileView` was never added. **This is a loose end**: `MapView` exists and works but has no way to reach it in the current build. Fix by adding a row/button in `ProfileView` similar to its existing "Settings" row.
11. **Profile** (`ProfileView`) — back chevron, Scout idle, username, stat chips, a "Settings" row (→ `SettingsView` via `.navigationDestination(for: SettingsDestination.self)`).
12. **Settings** (`SettingsView`) — custom top bar (back chevron + "Settings" title, NOT the system nav bar — see fix history, this used to double up with a system-provided back button), then a plain `List`: Username (row → alert with TextField, calls `authProvider.updateUsername(_:)` — **not** `signUp`, which would have created a duplicate `UserProfile` row each edit, this was caught and fixed during the build), Notifications toggle (local `@State` only, doesn't do anything yet — no local-notifications wiring implemented), Location (read-only status from `LocationProvider.authorizationStatus`), Help & contact (`mailto:` link), Delete account (confirmation alert → deletes all `CaughtDog` + `UserProfile` rows via `modelContext.delete(model:)`, calls `authProvider.signOut()`, dismisses — this drops the user back into the Launch/Landing/Onboarding flow since `RootView` will see `currentUsername == nil` again).

## Data flow for a catch (the core loop)

1. User taps "Catch a doggo" on `CollectionView` → `openCamera()` animates `surfaceState = .camera` (the shared `matchedGeometryEffect` id carries the pill's marigold background shape into the camera panel's black background shape — see decision #5).
2. `CameraView` appears as the panel's crossfading content layer, `.task` calls `CameraViewModel.start()` → requests camera permission, configures `AVCaptureSession` (no-ops gracefully if no hardware/permission — `CameraService.hasCameraInput` guards against calling `capturePhoto` with no configured input, which used to hard-crash the app on Simulator before this was fixed).
3. User taps the shutter → `CameraView.handleShutter()` → `viewModel.attemptCatch(in: modelContext)`:
   - Plays the whistle, waits 180ms (so the chirp is heard before "capture").
   - Calls `cameraService.capturePhoto()`. If it returns nil (no real camera — always true on Simulator) **and** the app is running in the Simulator (`#if targetEnvironment(simulator)`), a solid-color placeholder `UIImage` is used instead **and the catch is treated as automatically successful** — a deliberate dev/testing affordance so the whole flow can be exercised without real hardware; on a real device with a real photo, the photo is run through `DogDetector.detectSubject(in:)` and the catch only succeeds if Vision actually finds a dog.
   - On success: reverse-geocodes + coarsens the current location (or falls back to "Somewhere nearby" if location isn't available/authorized), generates a name/breed-label/traits via `CatchNameGenerator`, computes the next serial number via `modelContext.fetchCount(FetchDescriptor<CaughtDog>())`, inserts + saves a new `CaughtDog`.
4. `CameraView.onCaught(dog)` fires → `CollectionView.handleCaught(dog)` sets `caughtDog = dog` and animates `surfaceState = .celebration` directly (skipping back through `.idle`) — the shared `matchedGeometryEffect` id now carries the camera panel's black background shape straight into `CatchCelebrationView`'s `DoggoCardView`, so the whole chain (pill → viewfinder → card) reads as one continuous morph, not a hard cut to a modal.
5. `CatchCelebrationView` renders as a third inline peer state inside `CollectionView`'s own `ZStack` (not a `.fullScreenCover` — modals can't share `matchedGeometryEffect` across the presentation boundary). Tapping "Add to pack" calls the passed-in `onAddToPack` closure → `CollectionView.returnToIdle()`, morphing the card back down into the pill (the dog is already saved, so this is purely a visual return to idle); "Share instead" opens `ShareView` as a `.sheet` on top (sheets are unaffected — they don't need to share the geometry namespace).

## Build & run — IMPORTANT toolchain note

This project's `project.pbxproj` has `objectVersion = 110` and was created with **Xcode 27 (a beta not yet the default `/Applications/Xcode.app` on this machine)**. The regular `xcodebuild` (Xcode 26.5, at `/Applications/Xcode.app`) refuses to even open the project ("future Xcode project file format"). You **must** use:

```bash
export DEVELOPER_DIR=/Applications/Xcode-beta.app/Contents/Developer
xcodebuild -project DoggoCollector.xcodeproj -scheme DoggoCollector \
  -destination 'id=<simulator-udid>' build
```

Find a valid simulator UDID for this toolchain with:
```bash
export DEVELOPER_DIR=/Applications/Xcode-beta.app/Contents/Developer
xcodebuild -project DoggoCollector.xcodeproj -scheme DoggoCollector -showdestinations
```
(The Xcode-beta toolchain has its own simulator runtime list, separate from the regular Xcode's `xcrun simctl list` — an iPhone 17 Pro running iOS 27.0 was used throughout this build, UDID `F21032E2-9E21-4A53-87E3-54253EA3C18D` at time of writing, but UDIDs aren't stable across machines/resets — always re-discover it.)

To install/launch/screenshot after building:
```bash
APP_PATH=$(find ~/Library/Developer/Xcode/DerivedData -maxdepth 6 -name "DoggoCollector.app" -path "*iphonesimulator*" | head -1)
xcrun simctl install <udid> "$APP_PATH"
xcrun simctl launch <udid> com.DoggoCollector
xcrun simctl io <udid> screenshot /path/to/out.png
```

For **interactive** testing (tapping through the actual UI), plain `xcrun simctl` has no touch-injection command. Use the `computer-use` MCP tools (`request_access` for "Simulator", then `left_click`/`screenshot` etc. against the Simulator app window) — this requires the user to have granted Accessibility + Screen Recording permission to the Claude desktop app in macOS System Settings first (a mid-session permission change requires quitting/reopening the Claude app, which would kill an in-progress session — plan testing around this, don't ask the user to restart mid-task).

## Known issues (unresolved at time of writing)

1. **`CardDetailView`'s custom top-bar buttons (back chevron, heart favorite) do not respond to taps.** Reproduced reliably from a fresh app launch, independent of the Share sheet or any other prior interaction — it is the very first tap that fails. Other buttons on the exact same screen (Rename, Share this doggo) work fine.

   **Ruled out across two separate debugging sessions** (don't re-test these):
   - Not a `NavigationStack`/push-specific issue — switching `CardDetailView` from `.navigationDestination(for: CaughtDog.self)` to `.fullScreenCover(item:)` made no difference; the buttons stayed broken under a modal presentation too. (This change was tried and then **reverted** — `CollectionView` is back to the `NavigationLink(value: dog)` / `.navigationDestination(for: CaughtDog.self)` push pattern.)
   - Not `.toolbar(.hidden, for: .navigationBar)` — removing it entirely made no difference.
   - Not the system edge-swipe-back gesture specifically vs. the custom button — the user confirmed **both** are broken together, which was the clue that ruled out a simple "this one button's hit-testing is offset" theory.
   - A `log stream` capture during a failed tap showed the touch event marked `shouldSend: 0` with `systemGestureStateChange: 1` — iOS's touch dispatch classified it as a system gesture rather than delivering it to the SwiftUI button.

   **Isolated via binary search** (comment out content, rebuild, retest, repeat — this is the productive technique here, not console log reading): with `topBar` as the *only* content in the view (background + back/heart buttons, nothing else), **the buttons work fine**. Adding `DoggoCardView` back below `topBar` reproduces the bug. Continuing to strip `DoggoCardView` down: the bug survives with **just the photo `Image(uiImage:).resizable().scaledToFill().frame(height:260).clipped()`, no overlays, no background/clipShape/shadow, no bottom info section**. `Canvas`/`PolkaDotPlaceholder` is *not* the cause — the test dogs used during isolation had real `imageData` (even the Simulator fallback capture produces a solid-color `UIImage`, so the `Image(uiImage:)` branch was always the active one, not the `Canvas`-based placeholder branch. The binary search was interrupted (session/context constraints) one step before confirming whether the bare `Image(uiImage:)` view alone (no `.clipped()`, no `.frame`) reproduces it, or whether it needs `.scaledToFill()` + `.clipped()` specifically.

   **Next step for whoever picks this up**: resume the binary search inside `DoggoCardView`'s `photo` computed property specifically — try the image with only `.resizable()` (no `.scaledToFill()`/`.clipped()`), then add each modifier back one at a time, to find which exact modifier on the `Image` view is swallowing touches elsewhere on screen. Given the symptom (a system gesture recognizer classification, not a normal miss), also worth testing whether this is a `.clipped()`-specific interaction (clipping large image content sometimes changes a view's effective hit-testing bounds calculation in has-been-buggy ways across SwiftUI versions) — try `.frame(height: 260).clipped()` without `.scaledToFill()` as one data point. This is almost certainly a real bug in this specific iOS 27 beta / Xcode 27 beta SwiftUI implementation given how narrowly it's scoped (one specific view subtree, one specific rendering combination) rather than anything conceptually wrong in the app's code — attaching Xcode's live view debugger to a real running instance (not remote screenshot-based automation) would likely resolve this in minutes by showing the actual hit-testing view hierarchy, which remote automation cannot do. See task tracked as "Investigate CardDetailView top-bar button unresponsiveness."
2. **`MapView` has no navigation entry point.** It was reachable from a header icon on `CollectionView` that was removed when matching the real design (which only has one avatar icon there). Needs a new entry point, most naturally a row inside `ProfileView` next to "Settings".
3. **Onboarding screen is unverified against the real design** (see Source Documents). Currently a reasonable extrapolation, not a confirmed match.
4. **Baloo 2 / Nunito Sans fonts are not embedded** — using SF Rounded as a placeholder. See decision #2 for exactly how far the font-embedding investigation got and what to do next.
5. **`CLGeocoder` is deprecated as of iOS 26** (compiler warning in `LocationTagger.swift`) — still functionally works on this toolchain, but should migrate to MapKit's newer geocoding API eventually.
6. **Notifications toggle in Settings is inert** — no local-notifications permission request or scheduling wired up yet.
7. **"Save photo" / "Copy link" transient confirmation text** (in `ShareView`) — the underlying logic (flip a `@State` bool true, sleep 1.6s, flip false) is straightforward and was visually confirmed to work once, but is timing-sensitive to verify via remote screenshot automation (round-trip latency between click and screenshot regularly exceeds the 1.6s window). Not believed to be buggy, just hard to screenshot mid-transition.
8. **A stray `nonisolated(unsafe)` on `CameraService.hasCameraInput`** produces a harmless "has no effect" warning under this project's current concurrency settings; removing it entirely (tried during the build) surfaces a different, real warning about `CameraService`'s `AVCapturePhotoCaptureDelegate` conformance being MainActor-isolated while called from a background queue (flagged as "an error in Swift 6 language mode"). Currently left as a warning, not a build failure, on this toolchain. If this project ever moves to strict Swift 6 mode, `CameraService`'s delegate-callback threading will need a real redesign (e.g., explicit `nonisolated` delegate methods that hop to MainActor via `Task`, or restructuring capture to avoid the delegate pattern).

## What's explicitly out of scope for v1 (per the original build spec — don't build toward these unprompted)

No rarity/stats/battle mechanics, no in-app currency, no push notifications backend, no OAuth, no server-side account system. The `AuthProviding` protocol is the only deliberate concession for future extensibility.
