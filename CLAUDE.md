# DoggoCollector — Project Context for Claude

This file exists so a fresh Claude Code session (new chat, cleared context window) can pick up this project with zero prior conversation history and not get confused. Read this fully before making changes.

## What this app is

An iOS app (SwiftUI, iOS 27 target) where you point your camera at a real dog you meet, "catch" it, and it becomes a collectible card in your personal album ("Your Pack"). First app in a planned "Collector" family (Doggo → Cat → Flower), each with its own subject/hook, sharing a common Core/DesignSystem layer.

**Visual identity: "Sunny Fetch"** — marigold + cream palette, rounded storybook-ish shapes, a mascot named **Scout** (a simple-shapes dog head), playful/bouncy tone. Not calm/boutique/minimal — the design brief explicitly pushes against "generic AI template" looks (warm-cream+serif, near-black+neon-accent, newspaper-with-hairlines).

## Key product decisions already made (don't re-litigate these)

1. **No walk-streak mechanic.** The build spec originally said Doggo's hook was a walk-streak, calling it "already decided." But the actual design process (visible in the design tool's chat log) shows the user explicitly told Claude Design to drop streak for "collector-first... just capture real dogs," with sharing as a must-have. Confirmed with the user directly in this project too. The home screen is "Your Pack" — a collection grid + catch stats (caught/breeds/today) — not a streak dashboard. `Mechanic/PackCollectorMechanic.swift` implements this.
2. **Typography: SF Rounded, not Baloo 2 / Nunito Sans (for now).** The design's actual brand fonts are Baloo 2 (display) + Nunito Sans (body), both Google Fonts. Font embedding was attempted (downloaded both variable-font files, inspected their `fvar` tables with fontTools) but abandoned mid-implementation at the user's request ("if font is getting tricky, skip for now, we'll add fonts in phase 2"). Currently `DesignSystem/Typography.swift` uses `.system(size:weight:design: .rounded)` everywhere via the `DoggoFont`/`DoggoTextStyle` enums. **Phase 2 TODO**: re-embed Baloo 2 + Nunito Sans. Notes for that work: Baloo 2's variable font has clean named-instance PostScript names (`Baloo2-Regular/Medium/SemiBold/Bold/ExtraBold`) so `Font.custom(name:size:)` should work directly once registered via `CTFontManagerRegisterGraphicsFont`. Nunito Sans's variable font does NOT have per-instance PostScript names (multi-axis: wght/wdth/opsz/YTLC) — that one needs the `UIFontDescriptor` + `.traits: [.weight: ...]` approach instead, or find static per-weight files.
3. **~~No custom breed classifier~~ — SUPERSEDED by decision #15.** `DogDetector` (Vision `VNRecognizeAnimalsRequest`) still only confirms "a dog is somewhere in this photo," but a real trained breed classifier (`CoreMLBreedClassifier`) now runs alongside it at catch time — see decision #15. Names stay generated/editable regardless (**this part is unchanged**): catches get a **default generated name that is editable**, not silently assigned forever. See `Mechanic/CatchNameGenerator.swift` (name/traits — breed-label generation is no longer used for new catches, see decision #15) and the "Rename" affordance in both `CatchCelebrationView` and `CardDetailView` (tap pencil icon → alert with a TextField → mutates `dog.name` directly, autosaves via SwiftData).
4. **The share "second person collects this dog into their account" viral loop is NOT implemented.** Phase 1 auth is local-only (no backend, no server accounts), so there's nowhere for a second person's tap to land. `Core/Sharing/ShareLinkBuilder.swift` now builds only the outgoing share caption (Instagram Story sticker + native share sheet still work off the rendered card image) — the old `doggocollector://catch/<uuid>` deep link and fake `doggocollector.app` display URL were removed in decision #22 (the scheme was never registered or handled, so the links were dead strings). The receiving end + a real universal link are a Phase 2 item alongside Firebase + Sign in with Apple/Google.
5. **Camera morph must NOT be full-screen, must use the "peer-layer" matchedGeometryEffect pattern, and now extends all the way into the Gotcha card.** Initial build made the camera cover the entire screen (matching a literal reading of the design chat's "morphs into a full-screen rounded viewfinder" line); a first fix made it a contained bottom panel but applied `matchedGeometryEffect`/`clipShape` directly to the whole `CameraView` (including its live `AVCaptureVideoPreviewLayer`) driven by manually-tracked `topRadius`/`bottomRadius` `@State` — the user tested this and said the morph "looks very bad." The user then supplied a working reference SwiftUI file (a ChatGPT-camera-menu clone) demonstrating the correct technique, and the fix was rebuilt to match it exactly, then **extended one step further** at the user's request so the shutter tap continues the same morph straight into the caught dog's card, rather than cutting to a `.fullScreenCover`.

   **The pattern**: matchedGeometryEffect goes on a trivial background shape (`Color.black`/`DoggoColor.marigold`/the card itself + `.clipShape(...)`) — never on complex UIKit-backed content. Each state is its own small "box" view with a **fixed** frame/shape for that state; matchedGeometryEffect interpolates the frame automatically between whichever two fixed geometries are adjacent in a transition, so there's no need to track a continuous corner-radius value by hand. Actual content (icon+text for the pill; the whole live `CameraView` for the panel) sits as a **separate peer layer inside the same ZStack**, tagged only with `.transition(.opacity)` — it crossfades independently of the geometry animation instead of being dragged through it, which is what made the UIKit camera preview look broken originally. Each conditional box is wrapped in its own stable `ZStack { if condition { box } }` anchor to stop SwiftUI sliding it in from a screen edge on appearance. Animation curve: `.spring(response: 0.4, dampingFraction: 0.8, blendDuration: 0)`, taken directly from the reference.

   **The chain is now 3 states, not 2**: `CollectionView` has a private `SurfaceState` enum (`.idle` / `.camera` / `.celebration`), replacing the old `isCameraActive: Bool` + `.fullScreenCover(item: $justCaught)`. `CatchCelebrationView` is no longer presented modally — it's rendered as a third peer anchor inside `CollectionView`'s own `ZStack`, because **matchedGeometryEffect cannot cross a `.fullScreenCover`/`.sheet` presentation boundary** (a modal is a separate view-controller/window context; the shared `Namespace.ID` has no continuity into it). `CatchCelebrationView` now takes `morphNamespace: Namespace.ID` and `onAddToPack: () -> Void` as parameters instead of reading `@Environment(\.dismiss)`, and applies `.matchedGeometryEffect(id: "catchSurface", in: morphNamespace)` directly to its `DoggoCardView` (unlike the camera panel, the card's own SwiftUI content is simple enough — no UIKit bridging — to be the geometry-matched element directly, rather than needing a separate synthetic background layer). `CameraViewModel.attemptCatch`'s success callback (`CameraView.onCaught`) now flows into `CollectionView.handleCaught(_:)`, which sets `caughtDog` and animates `surfaceState` straight to `.celebration` — the camera panel is never explicitly closed first, so the black rounded rect morphs directly into the white card rect in one continuous animated transaction. "Add to pack" calls `onAddToPack` → `returnToIdle()`, which reuses the same shared id to morph the card back down into the pill, symmetrically, for free. See `CollectionView.swift`'s `SurfaceState`, `catchButton`, `cameraPanel`, `handleCaught`, `returnToIdle`, and `CatchCelebrationView.swift`.
6. **iPhone-only, portrait-only.** `TARGETED_DEVICE_FAMILY = 1`, `INFOPLIST_KEY_UISupportedInterfaceOrientations_iPhone = UIInterfaceOrientationPortrait` in the pbxproj build settings (both Debug/Release, main target). iPad support was deliberately removed from the default Xcode template.
7. **Nav restructure (from the care-community-flows brief): Collection's header avatar became a Map button; a new circular "paw" button opens Care; Profile moved one level down, nested under Care.** Previously Collection's header icon went straight to `ProfileView`. Now: `CollectionView.header`'s circular button is a `NavigationLink(value: MapDestination())` (50×50 marigold circle, white `mappin` glyph) instead of the old avatar/`ProfileDestination` link. Collection's bottom idle anchor changed from just `catchButton` to `HStack(spacing: DoggoSpacing.md) { catchButton; pawButton }` — `pawButton` (58×58 white circle, 1.5pt `#F3E4CC` border, marigold `pawprint.fill`) is a `NavigationLink(value: CareDestination())`. `catchButton` keeps flex-filling the remaining width for free (its background `Color` is greedy inside its `ZStack`, so `HStack` gives the fixed-size paw button 58pt and the rest to the button — no explicit `.frame(maxWidth:)` needed). `CareView` carries its own top-right Profile button (`NavigationLink(value: ProfileDestination())`, 44px white circle) — this resolves stack-wide against the `.navigationDestination(for: ProfileDestination.self)` already registered on `CollectionView`'s `NavigationStack` without re-registering anything, since `CareView` is pushed into that same stack. Net result: `Pack → (paw) Care → (top-right) Profile` and `Pack → (top-right, header) Map`. This also resolves the old "MapView has no navigation entry point" known issue for free (see Known Issues, now marked resolved). `CareDestination`/`MapDestination` are trivial `Hashable` marker structs declared next to the existing `ProfileDestination` at the bottom of `CollectionView.swift`.
8. **The 4 new flows (Insight Panel, Nearby Care, Neighborhood Pulse, Share metadata) all ship now as fully-built UI backed by mocked/local data — the real data sources are explicit, deliberate deferrals, not oversights.** Each sits behind a small protocol seam (matching the existing `AuthProviding`/`CollectorMechanic`/`SubjectDetecting` pattern) specifically so a real backend can drop in later without touching any view code:
   - **Flow 1 ("Scout's Sniff", Card Detail insight panel)**: breed guess / age bracket / care tips / did-you-know fact come from `Mechanic/DogInsightProviding.swift`'s `DogInsightProviding` protocol. **Now backed by a real implementation, `FoundationModelsInsightProvider`** (see decision #9) — `MockDogInsightProvider` (deterministic, seeded off `dog.id.hashValue` via `SeededGenerator` — same trick as `placeholderSeed`) still exists and is reused as the graceful-degradation fallback, not deleted. `InsightPanelView`/`ShareView` depend only on `DogInsightProviding`, never on either concrete type directly.
   - **Flow 2 (Nearby Care)**: originally both Vets and Shelters & NGOs came from `MockCareDirectory`. **Now resolved — both are backed by a real `MKLocalSearch` provider, `LiveCareDirectory`.** See decision #13.
   - **Flow 3 (Map's Neighborhood Pulse)**: locality aggregate counts come from `Core/Community/CommunityStatsProviding.swift`'s `CommunityStatsProviding` protocol, currently only `LocalCommunityStatsProvider` — which groups **this device's own local catches** by `locationLabel`. The brief's "48 dogs looked after around here this month" copy implies multi-user community data, which doesn't exist yet (Phase 1 has no backend at all, per decision #4) — the honest choice made here (confirmed with the user, who said a Firebase backend is planned but "just make the UI first") was to compute real, correct aggregates from the one data source that actually exists (local catches) rather than fabricate a multiplier. Numbers will look small/realistic for a single device today; that's expected, not a bug.
   - Care and Community's mocks lean on the same tiny `Core/SeededGenerator.swift` (`SeededGenerator: RandomNumberGenerator`, splitmix64-style) so mock content is stable across repeat visits instead of reshuffling on every render — deliberately not a shared "fake data" abstraction beyond that, since each domain's shape is different enough not to force one.
9. **Flow 1's insight generator is now real — a three-tier chain through Apple Intelligence via the iOS 27 Foundation Models framework — not mocked.** This framework changed substantially at WWDC 2026 (after Claude's training cutoff), so before implementing, the actual API surface was read directly off the installed Xcode-beta SDK's `.swiftinterface` files rather than trusted from training memory or web summaries alone — see **`AppleIntelligence.md`** at the project root for the full reference (exact API signatures for both model tiers, a UIKit cross-import gotcha for image attachments, error handling, PCC eligibility/quota constraints, and testing caveats — note `AppleIntelligence.md` itself still describes the original image-attachment design; the two sub-bullets below reflect what actually shipped after decision #15's text-only rewrite). `Mechanic/FoundationModelsInsightProvider.swift` conforms to the existing `DogInsightProviding` protocol via a **three-tier chain, on-device first**:
   1. **`SystemLanguageModel.default`** (on-device, no network) — checks `.availability`. **As of decision #15, this is text-only**: no photo attachment, no `.capabilities.contains(.vision)` check (text needs no vision capability). The prompt just states the breed (already known from `CoreMLBreedClassifier`, not guessed by the model) and asks for age bracket/care tips/fact, constrained to a private `@Generable` struct (age bracket via `GenerationGuide.anyOf`, care tips via `.count(3...4)`). If unavailable or it throws, escalates to tier 2 — there's no more confidence-based escalation, since the model no longer guesses breed (the thing it used to be "confident" or not about).
   2. **`PrivateCloudComputeLanguageModel()`** (Apple's server-side model) — **an explicit, deliberate exception to Phase 1's local-only/no-backend philosophy (decision #4), decided directly by the user** ("the backend is given by Apple in the form of `PrivateCloudComputeLanguageModel`, so might as well use it") after being walked through the tradeoffs (network dependency, per-user daily quota, and — most importantly — free access being gated on the developer account's App Store Small Business Program enrollment + under-2M-downloads status, with no paid tier beyond that). **Currently disabled at runtime** behind `FoundationModelsInsightProvider.hasPrivateCloudComputeEntitlement` (a `private static let`, `false`) — see Known Issues #10 for why: merely instantiating `PrivateCloudComputeLanguageModel` without the `com.apple.developer.private-cloud-compute` managed-capability entitlement (Apple-approval-gated, this project doesn't have it) crashes the whole process instantly, uncatchably, and this was confirmed for real on the user's device. When the flag is (eventually, if ever) true: checked for availability/quota (`!quotaUsage.isLimitReached`) before attempting (no more vision-capability check either, same reasoning as tier 1); its result is accepted unconditionally, since it's already the escalation path. A shared generic helper (`generate<Model: LanguageModel>(using:prompt:)`) builds the session/prompt for both tiers, since both conform to the same `LanguageModel` protocol.
   3. **`MockDogInsightProvider`'s age/tips/fact content** — final fallback if both model tiers are unavailable or fail (with tier 2 disabled, this is effectively the immediate fallback whenever tier 1 doesn't pan out). **As of decision #15, the mock's own `breedGuess`/`isConfident` are discarded even on this path** — breed always comes from the classifier, never from the mock or the language model. The UI never knows which tier actually answered.
   **Critically, Claude cannot verify the actual AI output** — the iOS Simulator doesn't reliably run live Foundation Models inference, so this was verified only for compile correctness (`xcodebuild` succeeds against both a Simulator destination and the user's real device, `id=00008150-000212A22185401C` / "Rajat's iPhone 17e", including codesigning). **The user tests actual model behavior on that physical device** (iOS 27 beta, Apple Intelligence/Siri AI enabled) — don't claim this feature "works" beyond "compiles and falls back gracefully" until the user confirms on-device.
10. **Phase 1.5 "Guardian Mode" — pledge to look after a specific street dog, get a living dossier + one-tap care log + printable Shelter Pass — implemented from a written plan (`~/Downloads/doggocollector-new-flows-COMPLETE.md`), with copy/colors pre-extracted from the design prototype's raw source rather than re-attempting the flaky design-tool automation.** Additive only — no nav restructure, no tab bar; hangs entirely off Card Detail and the Pack screen.
    - **No collar detection.** User decision: the "this might be someone's lost pet" idea stays a single subtle static caption line on the catch celebration screen (`CatchCelebrationView`) — no Vision work, no Foundation Models call, no `collarDetected` field anywhere.
    - **Data model**: `CaughtDog` gained `isWard`/`pledgedAt`/`wardStatusRaw`/`sterilizationRaw`/`dietaryProfile`/`behavioralQuirks` plus six `assignedClinic*` fields, all with **literal defaults on the property declarations** (required for SwiftData lightweight migration of the existing on-device store — confirmed working: the app launched clean over a store with 7 pre-existing catches with no migration error). New `Core/Persistence/CareEntry.swift` `@Model` (one row per logged interaction: type/note/timestamp), related via `@Relationship(deleteRule: .cascade, inverse: \CareEntry.dog)`. `Core/Persistence/GuardianTypes.swift` holds `WardStatus`/`SterilizationStatus`/`CareEntryType` enums with their display metadata (title/icon/colors) as computed vars, so every view reuses one source instead of re-switching on the raw case.

Verified end-to-end: archived a ward, confirmed it vanished from the Wards tab (empty state + "View Past Wards →" appeared), confirmed it showed up dimmed in Past Wards with a status `TagChip`, confirmed tapping it still opened the exact same dossier (clinic, vitals, care timeline all intact).

11. **Native `Picker` + `.pickerStyle(.segmented)` is unusable on this iOS 27 toolchain — replaced app-wide with a custom `SegmentedTabs` component.** The native segmented control renders as a translucent "Liquid Glass" material with white-on-white text against this app's cream background — confirmed visually (zoomed screenshot showed "All Catches"/"Guardian Wards" nearly unreadable, washed-out beige-on-cream with white text). Rather than patch just the one Guardian Wards picker that surfaced it, all four segmented pickers in the app were converted to `DesignSystem/Components/SegmentedTabs.swift` (generic over `Value: Hashable`, options as `[(value:label:)]`, styled to match the existing `StatChip` marigold-selected/white-unselected language): `CollectionView`'s All Catches/Guardian Wards, `CardDetailView`'s Scout's Sniff/Guardian Dossier, `CareView`'s Vets/Shelters & NGOs, `MapView`'s Mine/Neighborhood. This guarantees contrast regardless of OS material rendering and is also more consistent with the app's broader "everything is a bespoke SwiftUI view, nothing is an unstyled native control" pattern (`PillButton`/`TagChip`/`StatChip` already work this way) — the native segmented picker was the one exception. If a future session adds another 2-or-more-way tab control, use `SegmentedTabs`, not `Picker(...).pickerStyle(.segmented)`.
12. **A caught dog's photo is now square everywhere, app-wide — cropped once at capture, not re-cropped differently per screen.** Before this fix, the same stored photo appeared in at least three different, inconsistent landscape-ish crops depending on which screen displayed it (`DoggoCardView`'s old `.frame(height: isCompact ? 120 : 260)` fixed-height/flexible-width scheme meant compact grid tiles, the full Card Detail card, and `ShareView`'s separately-forced `.frame(height: 260)` preview each computed a different effective aspect ratio) — none of which matched the camera's own portrait-shaped viewfinder panel either. Root fix, all in this pass:

19. **Jetsam memory crashes (`EXC_RESOURCE`/"process exceeded a limit on resource consumption" kills — the likely explanation for Known Issue #19's "a lot of crashes at random actions") root-caused and fixed, implemented from a written audit (`~/Documents/memory_crash_fixes.md`) prompted by two concrete on-device repros: a shutter-tap catch, and a give-dose tap in Today's Care.** Both were confirmed jetsam memory kills, not logic crashes, and one root cause explains both — plus, very likely, most of Known Issue #19's broader unspecified reports.
    - **Root cause 1**: `CameraViewModel.croppedToSquare()` called `UIGraphicsImageRenderer(size:)` with no explicit format, which defaults to the *device screen scale* (3x on the user's iPhone 17e), not 1x. A 3024pt-square crop therefore rendered a 9072x9072px (~82MP) bitmap, and that's exactly what got JPEG-encoded and saved as `CaughtDog.imageData` — every catch made since decision #12 shipped square cropping stored an ~82MP photo.
    - **Root cause 2 (the multiplier)**: eight display sites across the app (Pack grid, Card Detail, Catch Celebration, Share, Today's Care avatars, Wards/Past-Wards thumbnails, medical-record thumbnails/pager, and `FoundationModelsInsightProvider`'s lazy breed-backfill) all decoded that stored data via plain `UIImage(data:)` — a full ~330MB decode each, even for a 34pt avatar circle. This also explains why the reports were "random" — baseline memory already sat near the cliff.

      1. `croppedToSquare(maxSide:)` (`Features/Camera/CameraViewModel.swift`) now pins `UIGraphicsImageRendererFormat().scale = 1` and caps output at 1080px — new catches store ~200-400KB JPEGs instead of ~82MP ones.
      2. `CameraViewModel.attemptCatch` now crops+downscales the captured photo *before* running `DogDetector`/`CoreMLBreedClassifier` on it (previously ran detection/classification on the full-res frame, with the crop happening last, right before save) — the full-res `UIImage` is released before Vision/Core ML/JPEG-encode buffers stack up. 3. **Onboarding** (`OnboardingView`) — **not verified against real design** (see Source Documents note above). ## Data flow for a catch (the core loop)

1. User taps "Catch a doggo" on `CollectionView` → `openCamera()` animates `surfaceState = .camera` (the shared `matchedGeometryEffect` id carries the pill's marigold background shape into the camera panel's black background shape — see decision #5).
2. `CameraView` appears as the panel's crossfading content layer, `.task` calls `CameraViewModel.start()` → requests camera permission, configures `AVCaptureSession` (no-ops gracefully if no hardware/permission — `CameraService.hasCameraInput` guards against calling `capturePhoto` with no configured input, which used to hard-crash the app on Simulator before this was fixed).
3. User taps the shutter → `CameraView.handleShutter()` → `viewModel.attemptCatch(in: modelContext)`:
   - Plays the whistle, waits 180ms (so the chirp is heard before "capture").
   - Calls `cameraService.capturePhoto()`. If it returns nil (no real camera — always true on Simulator) **and** the app is running in the Simulator (`#if targetEnvironment(simulator)`), a solid-color placeholder `UIImage` is used instead **and the catch is treated as automatically successful** — a deliberate dev/testing affordance so the whole flow can be exercised without real hardware; on a real device with a real photo, the photo is run through `DogDetector.detectSubject(in:)` and the catch only succeeds if Vision actually finds a dog.
   - Once a dog is confirmed present, `breedClassifier.classify(_:)` (decision #15) runs on the same frame — real photo or Simulator placeholder alike. A placeholder does NOT reliably come back "Indie mix" (see decision #15 — a solid-color image gets confidently misclassified as a specific breed instead), a known Simulator-only quirk that doesn't affect real device photos.
   - On success: reverse-geocodes + coarsens the current location (or falls back to "Somewhere nearby" if location isn't available/authorized), generates a name/traits via `CatchNameGenerator` (breed label comes from the classifier now, decision #15, falling back to `CatchNameGenerator`'s whimsical label only if classification fails), computes the next serial number via `modelContext.fetchCount(FetchDescriptor<CaughtDog>())`, inserts + saves a new `CaughtDog`.
4. `CameraView.onCaught(dog)` fires → `CollectionView.handleCaught(dog)` sets `caughtDog = dog` and animates `surfaceState = .celebration` directly (skipping back through `.idle`) — the shared `matchedGeometryEffect` id now carries the camera panel's black background shape straight into `CatchCelebrationView`'s `DoggoCardView`, so the whole chain (pill → viewfinder → card) reads as one continuous morph, not a hard cut to a modal.
5. `CatchCelebrationView` renders as a third inline peer state inside `CollectionView`'s own `ZStack` (not a `.fullScreenCover` — modals can't share `matchedGeometryEffect` across the presentation boundary). Tapping "Add to pack" calls the passed-in `onAddToPack` closure → `CollectionView.returnToIdle()`, morphing the card back down into the pill (the dog is already saved, so this is purely a visual return to idle); "Share instead" opens `ShareView` as a `.sheet` on top (sheets are unaffected — they don't need to share the geometry namespace).

## Build & run — IMPORTANT toolchain note

This project's `project.pbxproj` has `objectVersion = 110` and was created with **Xcode 27 (a beta not yet the default `/Applications/Xcode.app` on this machine)**. The regular `xcodebuild` (Xcode 26.5, at `/Applications/Xcode.app`) refuses to even open the project ("future Xcode project file format"). You **must** use:

```bash
export DEVELOPER_DIR=/Applications/Xcode-beta.app/Contents/Developer
xcodebuild -project DoggoCollector.xcodeproj -scheme DoggoCollector \
  -destination 'id=<simulator-udid>' build
```

Find a valid simulator UDID for this toolchain with:
```bash
export DEVELOPER_DIR=/Applications/Xcode-beta.app/Contents/Developer
xcodebuild -project DoggoCollector.xcodeproj -scheme DoggoCollector -showdestinations
```
(The Xcode-beta toolchain has its own simulator runtime list, separate from the regular Xcode's `xcrun simctl list` — an iPhone 17 Pro running iOS 27.0 was used throughout this build, UDID `F21032E2-9E21-4A53-87E3-54253EA3C18D` at time of writing, but UDIDs aren't stable across machines/resets — always re-discover it.)

To install/launch/screenshot after building:
```bash
APP_PATH=$(find ~/Library/Developer/Xcode/DerivedData -maxdepth 6 -name "DoggoCollector.app" -path "*iphonesimulator*" | head -1)
xcrun simctl install <udid> "$APP_PATH"
xcrun simctl launch <udid> com.DoggoCollector
xcrun simctl io <udid> screenshot /path/to/out.png
```

For **interactive** testing (tapping through the actual UI), plain `xcrun simctl` has no touch-injection command. Use the `computer-use` MCP tools (`request_access` for "Simulator", then `left_click`/`screenshot` etc. against the Simulator app window) — this requires the user to have granted Accessibility + Screen Recording permission to the Claude desktop app in macOS System Settings first (a mid-session permission change requires quitting/reopening the Claude app, which would kill an in-progress session — plan testing around this, don't ask the user to restart mid-task).

## Resolved since the last session

1. **`CardDetailView`'s custom top-bar buttons (back chevron, heart favorite) previously did not respond to taps — FIXED.** Two separate debugging sessions (documented at length in earlier revisions of this file, trimmed now that it's resolved) narrowed this down to `DoggoCardView`'s photo `Image` specifically, via binary search, but stalled one step before the actual root cause. **The user found the real fix**: add `.contentShape(Rectangle())` after `.clipped()` on the photo `Image` (`DesignSystem/Components/DoggoCardView.swift`, `photo` computed property). **Root cause / the general SwiftUI lesson**: `.clipped()` only affects *rendering* — it does not necessarily shrink a view's *hit-testing* region to match. An `Image` with `.resizable().scaledToFill()` can have a much larger pre-clip layout size than what's visually clipped to, and without an explicit `.contentShape(...)`, that oversized hit-testing region can silently extend over/compete with sibling interactive elements elsewhere in the same view hierarchy (here: `topBar`'s back/heart buttons, positioned above the photo in the same `VStack`, not even visually overlapping it) — which matches the `systemGestureStateChange: 1` touch-dispatch log captured during the original debugging. **Takeaway for future work**: whenever `.scaledToFill()` is used on an image that sits near other interactive elements, add `.contentShape(Rectangle())` (or otherwise clip the *container*, not just the image) as a matter of habit, not just when a bug shows up — this class of bug can affect siblings that don't even overlap the image visually.
2. **`MapView` had no navigation entry point — FIXED.** Resolved as a side effect of the care-community-flows nav restructure (decision #7): `CollectionView`'s header button now pushes `MapView` directly (`NavigationLink(value: MapDestination())`), replacing the old Profile-avatar link. No dedicated fix was needed beyond the restructure itself.
